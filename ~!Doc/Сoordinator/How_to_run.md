# 🎛️ Как запустить `workflow-coordinator`

**Дата создания**: 2026-04-14
**Файл агента**: `.claude/agents/workflow-coordinator.md`
**Модель**: `opus` (глубокий анализ решений)

---

## 🚀 Три способа запуска

### Вариант 1 — Явно назвать агента

В чате с Кодо пишешь:

```
Запусти workflow-coordinator. Продолжаем миграцию с репо spectrum, этап build.
```

Claude увидит имя `workflow-coordinator` → запустит его → агент прочитает контекст (MemoryBank) → спросит тебя детали.

---

### Вариант 2 — Описать задачу (Claude сам выберет)

```
Нужно продолжить миграцию DSP-GPU. Проведи через все этапы репо stats.
```

Claude по описанию цепочки `fix → build → test → doc` распознает что это работа координатора и позовёт `workflow-coordinator`.

---

### Вариант 3 — С конкретикой (рекомендуемый)

```
@workflow-coordinator

Репо: spectrum
Стартовая точка: прошёл fix, нужно build → test → doc
Ограничения:
  - на git push — спрашивай OK
  - на CMake изменения — спрашивай OK
  - на git tag — не ставь без явного разрешения
```

---

## 🎯 Как указать с каким репо работать

В промпте координатору — **явно назови репо и этап**.

### По одному репо

```
workflow-coordinator, возьми репо linalg и проведи полную цепочку
(fix → build → test → doc). На каждом critical OK — спрашивай меня.
```

### Цепочка нескольких репо

```
workflow-coordinator: запусти миграцию по графу зависимостей начиная с core.
После каждого репо — отчёт + жди OK перед следующим.
Порядок: core → spectrum+stats (параллельно) → signal_generators+linalg
       → heterodyne+radar → strategies → DSP
```

### Повторить один этап

```
workflow-coordinator: повтори только build-agent для репо heterodyne.
Не трогай fix и test — они уже пройдены.
```

### Проверить состояние без запуска

```
workflow-coordinator: прочитай MemoryBank/tasks/IN_PROGRESS.md и
покажи текущий статус миграции по 8 репо (не запускай никого).
```

## 🔄 Алгоритм работы координатора при старте

1. **Читает состояние** — `MemoryBank/tasks/IN_PROGRESS.md` и `MASTER_INDEX.md`
2. **Строит TodoWrite** на 8 репо × 4 этапа = **32 задачи**
3. **Спрашивает Alex**:
   ```
   Текущий статус:
     ✅ core — полностью готов
     ⏸ spectrum — прошёл fix, остановились перед build
   Варианты:
     A) Продолжить с spectrum/build
     B) Повторить spectrum/fix
     C) Переключиться на другой репо
     D) Другое
   Что делаем?
   ```
4. После твоего выбора — запускает подагент (`fix-agent` / `build-agent` / `test-agent` / `doc-agent`) через `Agent` tool
5. **Ждёт результата** → обновляет TodoWrite → переходит к следующему шагу
6. На `git push`, `git tag`, CMake Edit — **останавливается и передаёт запрос Alex**

---

## ⚠️ Правила работы координатора

| Что | Как |
|-----|-----|
| Файлы | ❌ Сам **НЕ пишет** `Edit`/`Write` — только `Read`/`Grep`/`Glob`/`Bash`/`TodoWrite`/`Agent` |
| Решения | ✅ Технические (какой агент запустить) — сам |
| Решения | 🔴 Push/tag/CMake-правки/удаления — **только с OK от Alex** |
| Прогресс | ✅ TodoWrite в реальном времени (после каждого этапа) |
| Ошибки | ⏸ Останавливается, показывает Alex, ждёт решения |
| Ветки | Работает только с `main` (ROCm/Debian/AMD). `nvidia` ветки нет. |

## 📊 Граф зависимостей (для подсказки)

```
core                            ← стартуем здесь (нет DSP-зависимостей)
  ↓
spectrum   stats                ← параллельно
  ↓         ↓
signal_generators    linalg     ← параллельно
  ↓                   ↓
heterodyne   radar              ← параллельно
  ↓
strategies                      ← зависит от всех выше
  ↓
DSP (мета-репо)                 ← финал
```


## 💡 Быстрый старт (diagnostics-режим)

Чтобы убедиться что координатор работает и понять текущее состояние, напиши:

```
@workflow-coordinator проверь статус миграции и покажи план
без запуска подагентов
```

Координатор только прочитает MemoryBank и выдаст статус по 8 репо. Ничего не изменит.
