## 3. ⚠ Противоречия / неточности

### 3.1. `factories.py` API изменился, но не задокументирован

В `factories.py` старые сигнатуры были `(gw, ctx)`, новые `(_gw=None, ctx=None)`. Я сохранила позиционную совместимость через `_gw`-игнорирование, но это **скрытая магия** — следующий разработчик прочитает `make_sig_gen(gw, ctx)` и подумает что `gw` ещё нужен.

**Действие**: либо переименовать функции (`make_numpy_sig_gen()`, `make_rocm_fft_proc(ctx)`), либо в docstring указать «`gw` параметр игнорируется, оставлен для обратной совместимости при миграции 2026-04-30».

### 3.2. `conftest.py` создан в `integration/` — но правило 04 говорит «conftest запрещён»

В `pytest_audit_2026-04-29.md`:

А я создала `integration/conftest.py` как proxy на factories.py. **Не для pytest** (внутри нет pytest-кода), а как имя для legacy-импортов `from integration.conftest import ...`.

- (A) Удалить `conftest.py`, поправить 2 теста на `from integration.factories import ...` — чище
- (B) Оставить, добавить **прямо в файле** комментарий «Это НЕ pytest conftest, это shim — см. правило 04»

Рекомендую **A** (~5 мин — поправить 2 импорта в `t_fft_integration.py` и `t_signal_gen_integration.py`).

После A5 я переписала `gpu_context.py` с `GPULoader.get()` на прямой `dsp_core`, но **никакой тест его не использует**. Я заменила все импорты на прямой `core.ROCmGPUContext(0)`.

## 5. 📅 Предложения для Phase B

### 5.1. Добавить B-Pre-build sanity (диагностика build)

Сейчас B0 проверяет файлы (✅ есть libs, нет shim, есть CMake-блок). Но не проверяет **build dependency tree**: например, `dsp_spectrum` зависит от `dsp_core` через `find_package(DspCore)` — если core не собрался, spectrum упадёт с непонятной ошибкой.

**Предложение**: в B0 добавить «test build` через `cmake --build --target dsp_core` отдельно сначала — диагностика по слоям.

### 5.2. Список тестов **с ожидаемым результатом**

Для B3 (полный прогон) сейчас просто `for test in t_*.py; do python3 ...; done`. Но без **ожидаемых** результатов сложно понять «4 SkipTest — это норма или проблема».

**Предложение**: создать в Phase B новый файл `expected_results_2026-05-03.md` с табличкой:
- 4 heterodyne — ожидается PASS (или указать failure mode)
- 2 spectrum/t_ai_* — ожидается SkipTest (нет api_keys.json)
- linalg/t_matrix_csv_comparison — ожидается PASS (data файлы скопированы)
- т.д.

После прогона — diff между expected vs actual покажет реальные проблемы.

### 5.3. Heterodyne acceptance — конкретные числа

Сейчас в Phase B про heterodyne:
> Если FAIL по точности — подкрутить `atol`

Но **не указано**:
- Что именно сравниваем (`expected_f_beat` vs `actual_f_beat` в t_heterodyne)?
- Какие текущие значения tolerance в коде (`F_BEAT_TOL_HZ = 5000.0`)?
- До какого предела поднимать atol — 5×, 10×, бесконечно?

**Предложение**: в TASK Phase B добавить «Heterodyne acceptance» секцию с конкретными числами из t_heterodyne.py.

### 5.4. Резервный план если CMake POST_BUILD не сработал

Phase B B1 говорит «если libs/ пустая → проверить логи». Но не описано **fallback**: что делать если POST_BUILD реально не работает на Debian (например, на старом CMake)?

**Предложение**: добавить B1.5 **fallback** = ручной `cmake --install build --prefix DSP/Python` — этот путь работал в legacy. Если POST_BUILD падает — запустить fallback и перейти к тестам.

### 5.5. Запоминание известных перспективных issues

Если в Phase B найдём pybind issues (нужна правка C++) — текущий план говорит «создать новый таск `TASK_pybind_fixes_<date>.md`». Лучше **сразу** в Phase B иметь:
- `MemoryBank/.future/TASK_pybind_review.md` (заготовка)
- В неё дописывать issues по мере находок

Не создавать новый таск **если** там просто 1-2 issues.

### 6.2. Шаблонная работа делается через `replace_all`

54 файла мигрированы за 3 ч потому что я использовала `Edit replace_all=true` для:
- `gpuworklib.X` → `core.X` или `spectrum.X`
- `gpuworklib.ROCmGPUContext` → `core.ROCmGPUContext`

Без replace_all это было бы 8-10 ч ручной работы.

### 6.3. Sanity check после **каждого** файла

После каждой миграции — `grep gpuworklib` + `python -c "ast.parse(...)"`. Это поймало 2 проблемы:
- Legacy путь в комментарии (`/home/alex/C++/GPUWorkLib`) — Alex заметил вовремя
- shadowing `spectrum` (модуль vs переменная) в t_hybrid_backend.py

Если делать sanity «в конце пакетом» — потеря в локализации причины.

### 6.4. CMake-патч 8 файлов через Read+Edit, не sed

Сначала пыталась удалить tail+append через sed → CRLF проблемы на Windows. Read + Edit (точечно после `install(...)` строки) — надёжнее.
