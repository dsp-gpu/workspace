# Честность Qwen (anti-hallucination): training vs inference, и почему не писать «сразу в режиме ревью»

> **Вопрос Alex (2026-06-08):** у Claude есть механизм «не обманываю» — как добиться у Qwen?
> Через deep-review задача↔код? Тогда почему не писать код сразу в режиме глубокого ревью?
> **Метод:** research 21 web/arxiv источник (2023-2026). **Связь:** `llm_router_architecture_2026-06-07.md` §7 (pipeline generate→review→verify).

---

## 1. Прямой ответ

**«Не обманываю» = ~70% обучение (в весах) + ~30% inference-обвязка.** Полностью «как у Claude» на
Qwen **без переобучения — нельзя**. Но inference-частью берётся **бóльшая доля практического
эффекта ДЛЯ КОДА**, потому что главный источник лжи в коде — **выдуманные API/методы**, а это
ловится **внешней проверкой**, не «совестью» модели.

- Claude'у честность даёт Constitutional AI / RLHF (модель учат говорить «не знаю») — это **веса**, промптом «будь честным» не воспроизвести.
- **Корень проблемы (OpenAI 2509.04664):** обучение и бенчмарки **награждают уверенную догадку выше «не знаю»** → модель структурно «не знает, что не знает». Это обучающий перекос, не инференс.

## 2. Почему НЕ писать сразу «в режиме ревью» (inline self-critique) — ГЛАВНОЕ

**Само-критика той же моделью в одном контексте — УХУДШАЕТ.** Цифры:

| Эффект | Данные | Источник |
|--------|--------|----------|
| Intrinsic self-correction роняет точность | GPT-4 GSM8K 95.5→89 (−6.5пп); GPT-3.5 CSQA 75.8→**41.8** (−34пп); Llama-2 62→36.5 | 2310.01798 (ICLR'24) |
| Self-refine **усиливает** предвзятость к своему черновику | 6 LLM, судья предпочитает свои генерации | 2402.11436 (ACL'24) |
| Помогает не «повтор», а **смена контекста** | same-session повтор 21.7% < единичного 24.6% F1 | 2603.12123 |
| Overthinking: модель «передумывает» и рационализирует неверное | точность падает при reasoning > порога задачи | OptimalThinkingBench 2025 |

**Вердикт:** inline-режим = «та же сессия» → anchoring + sycophancy + correlated blind spots
(модель не видит того, «чего не знает, что не знает», и предвзята к своему черновику). **Отдельный
проход обязателен** и должен быть **cross-context** (свежая сессия), **cross-model** или **external**.

> Inline self-check полезен УЗКО — как дешёвый **первый фильтр**: «перечисли использованные API/символы + флаг уверенности» → список на внешнюю проверку. НЕ замена verify.

## 3. Механизмы по силе (для кода)

| Механизм | Эффект (цифры) | Стоимость | Код |
|----------|----------------|-----------|-----|
| **External verify** (компилятор/тесты/symbol-registry) | бьёт любую само-критику; runtime-трейс — сильнейший сигнал | sandbox | 🟢 максимум |
| **RAG / API-doc grounding** | валидные low-freq API +9-25пп (StarCoder2 24.7→49.4) | средне | 🟢 высокий |
| **Cross-model** (B ревьюит A) | критик на чужом > на своём; слабый критик ведёт сильного решателя | +модель | 🟢 высокий |
| **Cross-context review** (свежая сессия, скрыть авторство) | F1 28.6 vs 24.6 (+11пп критич.), sycophancy исчезает | +1 проход | 🟢 высокий, дёшево |
| **Abstention по uncertainty** | semantic/symbolic entropy корр −0.56; abstention → FP <0.02% | дёшево-средне | 🟡 по API-именам |
| **Inline self-critique** (как verify) | **−6…−34пп** | дёшево | 🔴 опасно соло |

⚠️ **token-logprob/perplexity всего ответа — бесполезны для кода** (корр ~−0.04). Работает только
**min-logprob по токенам ИМЕНИ API** или **semantic entropy** (5-10× сэмплов).

## 4. Решение для нашего pipeline (`generate → review → verify`)

Правки по приоритету (impact для кода):

1. **External verify = must-have.** Каждый упомянутый класс/метод/FQN → прогон через **`dsp-asst` MCP** (`dsp_find`/`dsp_show_symbol`) + компиляция/линт + self-тесты в sandbox. Выдуманные DSP-GPU символы режутся **детерминированно** (главная защита: 19.7% генераций ссылаются на несуществующие пакеты, 58-84% выдуманных API без проверки).
2. **Deep-review = cross-context, НЕ inline.** Ревьюеру дать только {задача + финальный код}, **скрыть авторство и chain-of-thought** генератора, свежая сессия. +4пп F1, +11пп критич., sycophancy ↓.
3. **Cross-model.** **Qwen3.6-35B-MTP ревьюит код Qwen2.5-Coder-14B** (не сам себя). Критик на чужом ловит больше; MTP-сервер быстрый → латентность ок.
4. **Abstention по логпробу ИМЁН API** (не всего ответа): min-prob токенов имени < порога → форс-проверка через `dsp-asst` или явное «не уверен в этом API». Общая неуверенность — semantic entropy только на сложных кусках.
5. **Inline self-check — оставить как дешёвый первый фильтр** (список API+флаг уверенности → шаги 1,4). Сам по себе НЕ гарантия.
6. **Не давать «передумывать»:** ограничить длину reasoning / число итераций review (overthinking роняет точность). Один cross-context review + external verify > три self-refine.

**Итог:** честность Qwen придёт **не от промпта «будь честным»**, а от детерминированной внешней
проверки символов/компиляции (`dsp-asst` registry + sandbox) + cross-context/cross-model ревью +
abstention по логпробу имён API. Это воспроизводит основную долю эффекта Claude для кода **без
переобучения**. (Полную «совесть» — только дообучением на honesty/abstention, отдельная задача.)
