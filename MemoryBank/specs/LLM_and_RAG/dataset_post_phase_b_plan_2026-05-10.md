## ⏰ После Phase B (12.05) — нужны результаты обучения

### B1 — 3-way inference compare ⭐ (главный приоритет)

**Что:** запустить **3 версии модели** на одних и тех же 6 вопросах (как в smoke 2080 Ti) и сравнить ответы.

| Версия | Что | Откуда |
|--------|-----|--------|
| **baseline** | голая `Qwen3-8B` без обучения | huggingface as-is |
| **smoke** | модель после короткого теста на 2080 Ti (10.05, ✅ PASSED) | local checkpoint smoke_2080ti_2026-05-10 |
| **full** | модель после полного обучения 12.05 на RX 9070 (3 эпохи + r=16 + bf16) | local checkpoint phase_b_2026-05-12 |

**Зачем:** увидеть **что улучшилось**, а что **ещё ломается**. Например:
- baseline: 4/6 ошибок (выдумывает RochesterGPU, путает паттерны).
- smoke: 2/6 ошибок (улучшилось naming, осталась путаница Singleton/Bridge).
- full: 0/6? 1/6? Какие именно конкретные дыры остались?

**Когда:** сразу после 12.05, ~30 минут на прогон + анализ.

**Скрипт:** `C:/finetune-env/inference_compare.py` (уже есть, использовался для smoke).

### B2 — точечные anti-galлюц под top-3 weak spots ⭐

**Что:** взять **3 главные оставшиеся проблемы** из B1 и добить под них пары.

**Пример:** если B1 покажет что модель путает HIP-stream с CUDA-stream → сделать 50-100 пар именно про hipStream API. Если путает `ConsoleOutput::PrintLn` overloads → пары про сигнатуры.

**Зачем:** dataset уже большой (5777), но **не знаем заранее** какие конкретно дыры остались. Нужно **сначала измерить** B1, потом лечить точечно.

**Когда:** после B1, ~3-5ч.

### B3 — RAGAs faithfulness / abstain

**Что:** автоматическая метрика **«насколько модель опирается на RAG-контекст, а не на фантазию»**.

**Принцип:** даём модели вопрос **с RAG-контекстом** (выдержка из реального кода). Спрашиваем у судьи (другая LLM): «Содержится ли ответ в этом контексте? 0-1». Если 1 — `faithfulness=1.0`. Если 0 — модель выдумала.

**Зачем:** в проде модель работает с RAG. Если она игнорирует контекст и галлюцинирует — RAG бесполезен.

**Что нужно:** обученная модель в **Ollama** (locally) — чтобы прогнать N тестов автоматически.

**Когда:** после деплоя в Ollama, ~1-2ч.

**Партиально сделано:** `EV.E2` уже даёт RAGAs на golden-set v1 (`MemoryBank/specs/LLM_and_RAG/finetune_diagnostic_2026-05-08.md`). Расширение — golden-set v2 + abstain детектор.

### C1 — AI-summary через `qwen3-8b-dsp` (нашу!) на топ-44 классов

**Что:** взять **нашу обученную модель** после 12.05 и попросить **сгенерировать полное описание** для каждого из 44 ключевых классов (DrvGPU, FFTProcessor, CaponProcessor, ...).

**Зачем:** `class_facts` (37 пар, deterministic) — это **факты из БД** (имя/методы/brief). `class_summary` — **развёрнутый текст** ~500 слов: «класс X решает задачу Y, использует паттерн Z, ключевая идея — W».

**Прошлый опыт:** через **чистый** `qwen3:8b` галлюцинировал (2/5 ошибок: Singleton↔RAII путал, выдумывал кэширование). Поэтому отказались.

**Идея:** наша обученная модель **знает проект** — на тех же 44 классах не должна галлюцинировать. Если выйдет чистый — +44 богатых пары.

**Когда:** после Phase B, ~3-5ч + ручной deep-review каждой summary.

**Риск:** если модель ещё галлюцинирует — нужен жёсткий фильтр (deep-review каждого summary до коммита).

## ❌ Skip навсегда / опционально

### C2 — libclang cross-deps graph

**Что:** статический парсинг C++ через `libclang` (LLVM compiler frontend) → построить граф вызовов «метод A вызывает метод B».

**Зачем:** dataset научился бы отвечать «как X использует Y» через **точный** граф вызовов.

**Почему skip:** **9-11 часов работы** — нужно поставить libclang Python bindings, написать AST-обходчик, обработать 600+ файлов. До 12.05 не успеем, после — B-трек ценнее.

**Можно вернуться:** Phase C+ если будет нужно cross-class знание.

---

### C3 — Late Chunking BGE-M3

**Что:** улучшение **retrieval** (поиска контекста для RAG), не обучения. Современный embedding-метод от Jina AI.

**Почему skip:** не имеет отношения к **обучающему dataset'у** — это про **поиск** в RAG-системе. Отдельная задача (`CTX7` в `IN_PROGRESS.md`), отложена до настройки `transformers==4.46` venv на AMD Radeon.

---

### E1 / E2 — Synthetic Q&A loop / Adversarial GPT-4

**Что:**
- **E1:** заставить **нашу** модель **генерировать новые Q&A**, потом обучить её на собственном выводе (recursive self-training).
- **E2:** платный API (GPT-4 / Claude) — пусть придумают «adversarial» вопросы (нарочно сложные, чтобы сломать модель).

**Почему skip:**
- **E1 опасно:** если модель ошибается → её ошибки попадают в новый dataset → следующая итерация ошибается сильнее (**model collapse**).
- **E2 дорого:** тысячи запросов к платному API + время на curation.

**Можно вернуться:** только если Phase B покажет **острую** нехватку и других вариантов нет. Очень строгий фильтр обязателен (deep-review каждой пары).
