# Prompts — каталог промптов для локальной AI

> **Аудитория:** локальная Qwen 8B (дома) / 32B (работа). Она **глупее** Кодо (Claude).
> Промпты должны быть **простыми, шаблонными, с явными примерами и форматом вывода**.

---

## 1. Правила написания промптов для слабой модели

| Делать | Не делать |
|--------|-----------|
| Один промпт = одна задача | Многозадачные «сделай тест и заодно doxygen» |
| Жёсткий шаблон вывода (JSON / маркдаун с заголовками) | Свободная форма «опиши как считаешь нужным» |
| Few-shot: 1-2 коротких примера ВВОД→ВЫВОД | Длинная теоретическая инструкция без примеров |
| Явные ограничения: «не используй pytest», «не используй std::cout» | Полагаться что модель помнит правила проекта |
| Простые слова, короткие фразы | Длинные сложноподчинённые предложения |
| Указать что делать при неопределённости («если не знаешь — пиши TODO») | Открытое окно для галлюцинаций |
| Указать длину ответа («не более 200 слов», «10 строк») | Без рамок длины |

## 2. Формат файла промпта

Каждый файл — markdown:

```markdown
# NNN — короткое имя задачи

## Цель
Одна фраза.

## Когда использовать
Какой режим / какая ситуация.

## Вход
Что подаётся в `{user_message}` или контексте RAG.

## Системный промпт
\`\`\`
Ты — помощник проекта DSP-GPU.
[…конкретные инструкции…]
\`\`\`

## Шаблон пользовательского сообщения
\`\`\`
[…шаблон с {плейсхолдерами}…]
\`\`\`

## Формат вывода
Точная структура. Если JSON — пример JSON. Если код — без пояснений вокруг.

## Пример
ВВОД: …
ВЫВОД: …

## Анти-паттерны (что модель НЕ должна делать)
- Не использовать pytest / @pytest.* / import pytest
- Не использовать std::cout / std::cerr / printf — только ConsoleOutput::GetInstance()
- Не использовать GPUProfiler — только ProfilingFacade
- Не выдумывать классы которых нет в контексте — лучше написать TODO
```

## 3. Нумерация

`NNN_имя_задачи.md` где `NNN` — три цифры.

| Диапазон | Что |
|----------|-----|
| 001-099 | Помощники-индексеры (читать код, извлекать структуры) |
| 100-199 | Объяснения (explain) |
| 200-299 | Поиск (find) |
| 300-399 | Тесты (test) |
| 400-499 | Doxygen / документация |
| 500-599 | Refactor |
| 600-699 | Agent (системные промпты для tool-calling) |
| 700-799 | Eval (LLM-judge) |

## 4. Использование

В коде ассистента:

```python
from dsp_assistant.llm.prompts import load_prompt

prompt = load_prompt("002_write_test_runner_style")
system = prompt.system
user = prompt.user_template.format(class_name="FFTProcessorROCm", ...)
```

В Continue / Claude Code — копируешь системный промпт в slash-команду или в правило.

В голом Ollama — `cat prompt.md | ollama run qwen3:8b`.

---

## 5. Тестирование промптов

Каждый промпт должен иметь **хотя бы один пример** в разделе «Пример». На этих примерах строится мини-eval:
- Подаём вход локальной Qwen 8B → проверяем что вывод соответствует «формату вывода».
- Если модель часто «уходит в свободную форму» — упрощаем промпт.

Пакет eval-тестов лежит в `C:\finetune-env\eval\prompt_tests\` (Phase 8).

- Промпты пронумерованы в **порядке выполнения** на одном классе/файле:
  001 (индексация) → 002/003/004 (тесты) → 005 (doxygen) → 006 (refactor) → ...
- Если в одной фазе **два промпта запускаются параллельно** — суффикс: `005_1_xxx.md` и `005_2_yyy.md`.
