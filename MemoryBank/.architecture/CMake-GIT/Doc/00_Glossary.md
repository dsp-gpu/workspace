## Git и репозитории

### git
**Для инженера** — распределённая система контроля версий, хранит историю изменений в виде DAG коммитов. Каждый клон содержит полную историю.

**Для новичка** — это «машина времени» для кода. Ты пишешь код, делаешь «снимки» (коммиты), и можешь в любой момент вернуться к любому снимку из прошлого, посмотреть кто что менял и почему. У каждого разработчика — своя полная копия истории на диске.

---

### Рабочая копия (working tree / working copy)
**Для инженера** — каталог с файлами проекта + скрытая папка `.git/`, содержащая историю. С файлами можно работать напрямую — редактировать, запускать, тестировать.

**Для новичка** — это обычная папка с кодом, с которой ты работаешь в редакторе. В ней же спрятана (папка `.git/`) «книга учёта» всех изменений. Когда пишешь «у меня в проекте» — имеешь в виду именно эту папку.

**Пример**: `E:\DSP-GPU\core\` — рабочая копия репо `github.com/dsp-gpu/core`.

---

### Bare репозиторий (bare repo)
**Для инженера** — git-репозиторий **без** рабочего дерева. Хранит только объекты (commits, trees, blobs) и refs. Каталог имеет расширение `.git`. Используется как transport-узел между клонами.

**Для новичка** — это как «шкаф-каталог» в библиотеке: в нём только карточки-ссылки на содержимое (commits), а не сами файлы. Ты не можешь открыть и почитать код внутри bare-репо напрямую — его нужно «вытащить» через `git clone`. Используется когда репо — это **точка синхронизации**, а не место где кто-то работает.

**Пример**: `E:\DSP-GPU\_release_git\smi100_core.git\` — это bare, на SMI100 тоже bare.

**Создание**: `git init --bare myrepo.git`

---

### Клон (clone)
**Для инженера** — создание новой локальной копии удалённого репо через `git clone <url>`. Копирует всю историю + делает checkout главной ветки в рабочее дерево.

**Для новичка** — «сделать себе копию всей библиотеки на свой ПК». После `git clone` у тебя на диске появляется папка с кодом + вся история правок.

---

### Remote
**Для инженера** — именованный URL удалённого репозитория. Стандартное имя — `origin`. В одном локальном репо может быть несколько remote-ов (`origin`, `upstream`, `smi100`).

**Для новичка** — это «адресная книга»: вместо того чтобы каждый раз писать длинный URL, ты говоришь git «пуш в origin» — а он знает куда. У нашего проекта обычно два remote-а: `origin` (github.com) и `smi100` (локальная сеть).

## Ветки, теги, идентификаторы

### SHA (commit hash)
**Для инженера** — SHA-1 хеш коммита, 40 символов. Уникально идентифицирует состояние репо в момент коммита. Обычно сокращается до 7-10 символов.

**Для новичка** — это «серийный номер» каждого снимка в истории. Выглядит как `abc123def456...`. Если два разных коммита имеют разный SHA — значит в них разное содержимое (даже если отличается один пробел).

---

### Branch (ветка)
**Для инженера** — именованный указатель на конкретный коммит, который **движется вперёд** при каждом новом коммите. Основная ветка обычно называется `main` или `master`.

**Для новичка** — это «закладка» на определённое состояние кода. Когда ты делаешь новый коммит — закладка автоматически переезжает на новый коммит. Разные ветки = параллельные линии развития (например, `main` — стабильный код, `breaking/new-api` — эксперимент).

---

### Tag (тег)
**Для инженера** — именованный указатель на конкретный коммит, который **не движется**. Используется для фиксации релизов (`v1.2.0`, `v2.0.0`). Бывает lightweight (просто указатель) и annotated (с сообщением + подписью).

**Для новичка** — это «печать с датой релиза», которую ты ставишь на определённый коммит и больше не трогаешь. `v1.2.0` — значит «вот этот снимок мы официально назвали версией 1.2.0». В отличие от ветки — тег после создания **не переезжает**.

**Правило DSP-GPU**: теги никогда не переписываются. `v1.0.0` → `v1.0.1` → `v1.1.0`. `git push --force` на тег — запрещён.

---

### detached HEAD
**Для инженера** — состояние, когда HEAD указывает на конкретный коммит вместо ветки. Возникает после `git checkout <sha>` или `git checkout <tag>`. Новые коммиты в этом состоянии **не принадлежат** никакой ветке и могут потеряться при переключении.

**Для новичка** — «я встал на конкретный снимок прошлого, но ни на одной закладке не стою». Если в этом состоянии что-то пишешь и коммитишь — работа может потеряться когда вернёшься на ветку. Не опасно если просто смотришь, опасно если пишешь код. `build/_deps/` у FetchContent часто в этом состоянии.

---

## Механизмы подключения кода

### Submodule
**Для инженера** — ссылка внутри одного git-репо на конкретный коммит другого репо. Файл `.gitmodules` описывает URL + путь. Обновляется через `git submodule update --init --recursive`.

**Для новичка** — это «подпапка, которая на самом деле отдельный проект». Например, `my-app/` содержит папку `libs/logger/`, которая сама — самостоятельный git-репо. Удобно, но путаница: нужно **отдельно** делать `git pull` для основного проекта и для каждого submodule.

**В DSP-GPU**: **не используем**. Предпочитаем FetchContent/vendor.

---

### FetchContent (CMake)
**Для инженера** — модуль CMake, который во время configure скачивает указанный git-репо в `build/_deps/<name>-src/` и добавляет его через `add_subdirectory`. Управляется через `FetchContent_Declare` + `FetchContent_MakeAvailable`.

**Для новичка** — «встроенный в CMake курьер». В CMakeLists.txt пишешь «мне нужна библиотека DspCore версии v1.2.0 с такого-то URL» → при запуске `cmake` курьер сам идёт и приносит код в папку `build/_deps/`. Ты в своём репо ничего про DspCore не коммитишь — только «заказ курьеру» в CMake.

**Ключевые параметры**:
- `GIT_REPOSITORY` — откуда качать
- `GIT_TAG` — какую версию (тег/ветка/SHA)
- `GIT_REMOTE_UPDATE_STRATEGY CHECKOUT` — каждый configure проверять обновления
- `SOURCE_DIR` — куда положить (по умолчанию `build/_deps/`)
- `FETCHCONTENT_SOURCE_DIR_<NAME>` — переопределить источник (напр. взять из локальной папки)

---

### Vendored (vendoring)
**Для инженера** — подход, когда исходники внешних зависимостей **закоммичены в свой репо**. Обычно в папку `vendor/` или `third_party/`. Обеспечивает reproducibility и self-containedness.

**Для новичка** — «взять копию чужой библиотеки и положить её к себе в проект». Плюсы: не зависишь от того что чужой сервер работает, можешь собрать через 10 лет, клон даёт тебе всё сразу. Минусы: твой репо раздувается, обновлять надо вручную.

**В DSP-GPU**: **основной режим** для Zone 2 (LP). `vendor/` лежит в git LP_x, обновляется автоматически CMake + Python-скриптом.

## Форматы передачи

### git archive
**Для инженера** — команда `git archive --format=tar.gz <tag>` создаёт tarball из указанного ref-а **без истории**. Только файлы на момент этого ref-а.

**Для новичка** — «распечатка» снимка библиотеки в виде обычного архива. История потерялась, но сами страницы на месте. Удобно когда нужно отдать кому-то кто не пользуется git-ом.

---

### git bundle
**Для инженера** — команда `git bundle create out.bundle --all` упаковывает все объекты + refs в один файл. Получатель делает `git clone out.bundle local-repo` и получает **полный** git-репо со всей историей.

**Для новичка** — «git-репо, упакованный в один файл». Нужно передать кому-то репо без сети? Создаёшь `.bundle`, отдаёшь на флешке, получатель делает `git clone` прямо из файла. История, ветки, теги — всё сохраняется.

---

### Tarball
**Для инженера** — архив `.tar` (опционально сжатый в `.tar.gz`, `.tar.bz2`, `.tar.xz`). Стандартный Unix-формат упаковки директории.

**Для новичка** — это обычный архив, как zip, только другой формат. Распаковал → получил папку. Никакого git, никакой истории — просто файлы.

## Release engineering

### Promotion (промотирование)
**Для инженера** — процесс переноса **уже протестированного** релиз-тега из dev-контура в prod-контур. У нас: из `github.com/dsp-gpu/<mod>` → `E:\DSP-GPU\_release_git\smi100_<mod>.git` → `/srv/smi100/smi100_<mod>.git`. Делается вручную через `promote_to_smi100.sh`.

**Для новичка** — «поднять товар из склада в витрину». Не всё что есть на складе (в github) попадает в магазин (SMI100) — только то что Alex проверил и одобрил. Процесс называется «промотирование».

---

### Release-only репо
**Для инженера** — git-репозиторий, в котором хранятся **только теги релизов**, без истории разработки. Ветки `main`/`dev` не публикуются. Pre-receive hook отклоняет push не в `refs/tags/*`.

**Для новичка** — «склад запечатанных коробок». В него не попадает «промежуточный черновик» или «незаконченный эксперимент» — только готовые отмаркированные версии. У нас такие все `smi100_*.git`.

---

### Mirror
**Для инженера** — репозиторий, который является точной копией другого (включая все refs). Обычно bare, обновляется через `git clone --mirror` + `git remote update`.

**Для новичка** — «точное зеркало другого репо». Что в одном — то и в другом. У нас `smi100_*.git` — **НЕ** зеркала github (не всё туда попадает, только промотированные теги). Термин используется в общих обсуждениях.

---

### Pin
**Для инженера** — установленный флаг `pinned: true` в `deps_state.json`, который сообщает `update_dsp.py --mode lp-refresh` «не обновляй этот модуль, оставь как есть». Используется для стабилизации версий во время release-candidate.

**Для новичка** — «заморозить» версию зависимости. Обычно при каждой сборке LP автоматически обновляется с SMI100. Но если ты готовишь релиз и не хочешь чтобы в середине приёмки прилетел новый core v2.0.0 — ставишь pin, и обновления для core игнорируются пока не снимешь pin.

**Команды** (`scripts/update_dsp.py`):
- `--pin core v1.2.0 --reason "RC stabilization"` — заморозить
- `--unpin core` — снять pin
- `--status` — показать что замороженно
- `--pin-all` / `--unpin-all` — разом

Подробно — [05_Refresh_Mechanics.md](05_Refresh_Mechanics.md#pin).

---

### deps_state.json
**Для инженера** — JSON-файл в корне LP_x, содержащий `sha + tag + pinned` для каждой зависимости. Коммитится в git LP_x. Используется как SSOT (single source of truth) о текущем состоянии зависимостей.

**Для новичка** — «паспорт сборки» LP_x. В нём записано «сейчас core версии такой-то, spectrum такой-то». Если клиент жалуется на баг через полгода — делаешь `git checkout` на нужный момент, паспорт показывает какие тогда были версии → воспроизводишь точно ту же сборку.

Формат:
```json
{
  "schema_version": 2,
  "updated": "2026-04-19T10:00:00Z",
  "repos": {
    "core":     { "sha": "abc...", "tag": "v1.2.0", "pinned": false },
    "spectrum": { "sha": "fed...", "tag": "v1.1.0", "pinned": true, "pin_reason": "RC stab" }
  }
}
```

---

### Manifest (`dsp_modules.json`)
**Для инженера** — JSON-файл в корне workspace, описывающий все модули DSP-GPU и их зависимости. Используется как SSOT для генерации `fetch_deps.cmake` в каждом репо.

**Для новичка** — «паспорт всего проекта»: какие вообще есть модули, кто от кого зависит. Если появился новый модуль — добавил строчку в manifest, дальше всё автоматом (CMake-код генерируется из него).

Подробно — в `cmake_git_distribution_spec_2026-04-18.md §C1`.

---

### Config.cmake (и *ConfigVersion.cmake)
**Для инженера** — CMake-файл, экспортируемый пакетом, содержит `find_dependency` + `include(<Pkg>Targets.cmake)`. Пара к нему — `<Pkg>ConfigVersion.cmake` (генерируется через `write_basic_package_version_file`) — описывает политику совместимости версий (`SameMajorVersion` / `ExactVersion`).

**Для новичка** — «этикетка-совместимость» на библиотеке. Когда ты просишь «мне DspCore >= 1.5» — CMake смотрит на эту этикетку, и если библиотека отвечает «я 1.2» — сразу скажет **на этапе configure** «не подходит, поставь 1.5+». Без этой этикетки ошибка появилась бы только на этапе линковки, и была бы загадочной.

## Сборка и reconfigure

### Clean build
**Для инженера** — полная очистка `build/` и запуск CMake с нуля: `rm -rf build && cmake --fresh --preset <name> && cmake --build build`. Гарантирует отсутствие устаревших объектников и стейл-кеша.

**Для новичка** — «сборка с полного нуля»: удаляем всё что было собрано раньше, запускаем заново. Долго? Да. Зато гарантированно нет старых мусорных файлов, которые могут дать странное поведение. Политика DSP-GPU — **всегда clean build** в Zone 2.

---

### Incremental build
**Для инженера** — пересборка только изменённых файлов на основе зависимостей, определённых билд-системой (Ninja/Make). Работает пока build-система верит что метаданные актуальны.

**Для новичка** — «собрать только то что изменилось». Быстро, но ненадёжно: если CMake не понял что один из deps изменился — получишь собранный бинарник со старым кодом. Поэтому в prod у нас **всегда clean build**, incremental — только во время активной разработки.

---

### Reconfigure (cmake reconfigure)
**Для инженера** — повторный запуск CMake configure без удаления build/. Триггерится автоматически если изменился `CMakeLists.txt` или файл в `CMAKE_CONFIGURE_DEPENDS`.

**Для новичка** — CMake пересматривает «что из чего собираем», не трогая уже собранное. Нужно когда ты поменял зависимости или CMakeLists.txt, но всё скомпилированное остаётся валидным.

---

### Layer 1 (v2 спеки)
**Для инженера** — паттерн в `cmake/version.cmake` каждого модуля: early-return если текущий git HEAD hash совпадает с закешированным в `.git_hash`. Предотвращает регенерацию `version.h` при каждом configure.

**Для новичка** — оптимизация: «если ничего не изменилось — не перегенерируй version.h каждый раз». Экономит секунды, но главное — предотвращает триггер incremental-rebuild когда на самом деле ничего не изменилось.

---

### Layer 2 (v2 спеки)
**Для инженера** — паттерн в `CMakeLists.txt` мета-репо: `set_property(DIRECTORY APPEND PROPERTY CMAKE_CONFIGURE_DEPENDS ...)` на `.git/index` и `.git/FETCH_HEAD` каждого отслеживаемого модуля. Ninja автоматически запускает `cmake reconfigure` при изменении этих файлов.

**Для новичка** — «CMake сам замечает когда ты сделал `git pull`». Без Layer 2 — после `git pull` в модуле надо вручную `cmake -B build`, иначе `cmake --build` соберёт старую версию. С Layer 2 — ninja замечает «hm, `.git/FETCH_HEAD` изменился — надо перезапустить CMake» и делает это автоматически.

## Специфика DSP-GPU

### Zone 0
**Для инженера** — публичный контур разработки: `github.com/dsp-gpu/*` + ПК Alex. Место где делается вся работа над кодом.

**Для новичка** — «мастерская», где Alex с помощниками (AI, разработчиками) пишут код. Интернет есть, GitHub есть, MemoryBank есть. Всё что касается разработки — здесь.

---

### Zone 1
**Для инженера** — транзитный узел: SMI100 (Debian, локальная сеть, без интернета). Хранит release-only `smi100_*.git`, принимает patches через `incoming/`. Раздаёт код по LAN.

**Для новичка** — «склад готовой продукции между мастерской и клиентами». На склад попадает только проверенный товар, клиенты (LP) забирают себе, иногда предлагают улучшения обратно.

---

### Zone 2
**Для инженера** — изолированный контур потребителей: N × LocalProject. Нет интернета, есть LAN к SMI100. Обновляется через FetchContent + Python `update_dsp.py`.

**Для новичка** — «клиентская сторона», где код используется для реальных продуктов. Интернета нет, GitHub недоступен. Всё через локальную сеть с SMI100.

---

### LP (LocalProject)
**Для инженера** — конкретный проект в Zone 2, собственный git-репозиторий команды. Имя обычно `LP_A`, `LP_B`, ..., `LP_N`. Содержит свой код + `vendor/` + `deps_state.json`.

**Для новичка** — конкретный проект конкретной команды: «LP_A» = «проект команды А». У каждой команды свой LP, свои выбранные модули DSP-GPU, свой репо.

---

### Dev-overlay
**Для инженера** — клон модуля в `~/LP_x/dev-overlays/<mod>/`, переопределяющий source через `FETCHCONTENT_SOURCE_DIR_DSP<MOD>`. Preset `zone2-dev-<mod>` активирует overlay. Папка в `.gitignore` LP_x.

**Для новичка** — «рабочий столик сбоку, где ты правишь модуль». Когда хочешь серьёзно попатчить модуль — клонируешь его в отдельную папку внутри LP_x (но в `.gitignore`, не попадает в git LP) и говоришь CMake «бери модуль отсюда, а не из vendor/». Правки переживают `rm -rf build`.

---

### R-normal / R-offline / R-patches
**Для инженера** — три режима работы LP:
- **R-normal** — обычный workflow: cmake → refresh → build, LAN к SMI100 есть
- **R-offline** — LP перенесён на изолированный ПК, собирается автономно из `vendor/`
- **R-patches** — LP шлёт ветки фиксов обратно в Alex через `incoming/`

## Требования проекта

### R1-R9
Перечислены в `10_Decisions_Log.md`. Это базовые требования:
- R1 — только проверенные версии в Zone 1
- R2 — clean build в Zone 2
- R3 — добавление модуля ≤ 5 мин
- R4 — reproducibility через 6+ месяцев
- R5 — внятные CMake-ошибки (не линкерные)
- R6 — MemoryBank/~!Doc НЕ в Zone 1
- R7 — N LP обновляются в своём темпе
- R8 — patches из LP могут вернуться
- R9 — нет интернета на SMI100/LP
