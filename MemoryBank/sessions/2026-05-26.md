## ⚙ Production setup (для повторного запуска)

### Запуск llama-server (RX 9070, gfx1201, 16 ГБ VRAM)

```bash
# Освободить VRAM (Ollama + всё что держит)
sudo systemctl stop ollama
sudo pkill -9 -f "llama-server" 2>/dev/null

# Запуск (partial offload — auto-fit, ~30 слоёв на GPU, остаток CPU)
cd /home/alex/llama.cpp
nohup ./build/bin/llama-server \
    -m /mnt/data/Qwen3.6-35B-A3B-MTP-GGUF/Qwen3.6-35B-A3B-UD-Q4_K_M.gguf \
    -c 4096 \
    -fa on \
    -np 1 \
    --spec-type draft-mtp \
    --spec-draft-n-max 2 \
    --reasoning off \
    --host 127.0.0.1 \
    --port 8080 \
    > /tmp/llama-server.log 2>&1 &
```

⚠ **Критичные флаги:**
- `--spec-type draft-mtp --spec-draft-n-max 2` — активирует MTP speculative decoding (без них модель работает но без MTP-buff)
- `--reasoning off` — подавляет thinking trace (иначе Qwen3.6 уходит в reasoning trace, content урезается)
- Без `-ngl N` — auto-fit подбирает оптимум (с явным `-ngl 30` падает OOM)
- `-c 4096` — контекст; больше = больше KV-cache (на 16ГБ нельзя `-c 32768`)

### API (OpenAI-compatible)

```bash
curl -s http://127.0.0.1:8080/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{"messages":[{"role":"user","content":"…"}],"temperature":0.3,"max_tokens":4000}'
```

Response: `choices[0].message.content` (+ `reasoning_content` пустой при `--reasoning off`).

## 🔍 Ключевые открытия

### 1. **llama-server даёт +0.5-1.0 quality на 30B** ⭐⭐⭐

Та же модель (30B-A3B), та же квантизация (Q4_K_M), но через llama-server:
- DSP-GPU: 4.33 → **4.67** (+0.34)
- pao-contrib: 3.83 → **4.83** (+1.0!)

**Причина:** в Ollama T2 review pao застрял в repeat-loop (`BUG #9-#37` повторял одно). В llama-server нет такой проблемы (есть `repeat_penalty 1.15` через CLI). По скорости тоже выигрывает: 12.3 → 39.6 tok/s (×3.2).

### 2. **30B llama-server догоняет MTP по quality** при сравнимой скорости

| Модель | Quality avg | Wall-time DSP-GPU |
|--------|------------:|------------------:|
| MTP llama-server | 4.83 | 115s |
| **30B llama-server** | **4.75** | **87s** ⭐ |
| MTP llama-server (pao) | 4.83 | 135s |
| **30B llama-server (pao)** | **4.83** | **80s** ⭐⭐ |

→ **30B llama-server (без MTP-setup) — отличная альтернатива MTP** для большинства задач: качество то же, иногда быстрее по wall-time, не нужен specialty GGUF.

### 3. **14B-DSP через llama-server: variability**

- DSP-GPU: 3.17 → 3.33 (marginal), но **correctness 3.0 → 3.67** (лучше fact-recall)
- pao-contrib: 3.17 → 3.83 (заметно лучше)
- T3 describe в llama-server дал всего **44 tokens** (один пункт из 5) — chat template / sampling Coder-14b в llama-server **режет** ответы короче. Compensating: T6 indexing стал длиннее.
- **Скорость ×5.5** в обоих случаях (7.7 → 43 tok/s)

→ 14B-DSP **всё ещё проигрывает** по quality (ждёт v7 train). Но скорость через llama-server сильно лучше.

### 4. **Никто из больших не определил Bridge для HybridBackend**

- 35B-Q8 Ollama: правильно никто не определил
- MTP llama-server: Composite ❌
- 30B llama-server: Facade ❌ (ближе чем Composite)
- 14B llama-server: что-то generic

→ Это **подтверждает важность v7 train с явными pattern-class парами** (43 HybridBackend+Bridge пары уже в датасете).
