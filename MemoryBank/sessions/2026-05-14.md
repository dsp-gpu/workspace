## 🔬 Главные технические инсайты

### A. RX 9070 + ROCm 7.2 + bnb 0.49.2 — рабочий стек для QLoRA, но с известными bugs

- **bnb 4-bit kernel под gfx1201**: падает в `csrc/ops.hip:83` уже на step 9–19 при `max_seq=1024 + adamw_8bit`. Workaround → `max_seq=512`, `adamw_torch`, `expandable_segments` env (хотя последний игнорируется на ROCm 7.2 — warning «not supported on this platform»).
- **HIP `is_nonzero/item` race** в HF Trainer (NaN-check / LR scheduler) → `illegal memory access` на ~step 50–120 независимо от bnb. Это **фундаментальный bug ROCm 7.2 на RDNA4**. Для full Phase B → `--save-steps 20` + `--continue-from-adapter` auto-resume.
- **Безопасный конфиг Plan-B** для smoke на 9070: `--bf16 --optim adamw_torch --max-seq-len 256–512 --lora-r 8 --lora-alpha 16` + env `PYTORCH_HIP_ALLOC_CONF=expandable_segments:True`.
- Все доп. GPU-приложения (Telegram/браузер/видео) во время train **запрещены** — GPU compositor конкурирует с HIP scheduler → race.

### B. Coder-7B > general-8B на DSP-GPU задачах
На тех же 1200 парах из dataset_v4: `coder-7b eval = 1.18` vs `qwen3-8b eval = 1.26` (Δ = -0.085). Подтверждает решение брать **Qwen2.5-Coder-14B-Instruct** как главную модель Phase 5.

### C. RAG-обогащение датасета даёт реальный прирост качества
v6 на step 50: eval = 1.345 (на 1310 samples) vs v4 1.44 (на 70 samples). v6 не только **обыгрывает** в абсолюте, но и оценка на ×18 более надёжной выборке. Гипотеза Phase 3 подтверждена.

### D. Lesson на будущее: eval_split для больших датасетов
При `dataset >= 9K` пар обязательно `--eval-split 0.02` (~180 примеров). Иначе каждый eval на 1310 samples занимает 6–10 минут wallclock (×18 от v4) и общий smoke выходит из бюджета.
