Используй агент module-doc-writer для модуля: $ARGUMENTS

Если модуль не указан — спроси какой документировать.

Алгоритм:
1. Прочитай Doc/Modules/heterodyne/Full.md как эталон формата
2. Обязательно просмотр директории Doc_Addition и MemoryBank на предмет поиска полезной информации по данному модулю и используемые там методы. Все полезное и ценное копировать в Full.md для лучшего понимания проблемы.
3. Определи путь к документации:
   - DrvGPU → код в `DrvGPU/`, документация в `Doc/DrvGPU/` (не в Doc/Modules!)
   - Остальные модули → код в `modules/$ARGUMENTS/`, документация в `Doc/Modules/$ARGUMENTS/`
4. Изучи реальный код модуля (include/src/kernels/tests)
5. Изучи python/py_$ARGUMENTS.hpp и Python_test/$ARGUMENTS/
6. Создай ДВА файла (оба обязательны!) если их нет. Если они есть редактируй — главное не потерять то что есть:
   - {doc_path}/Full.md — полная документация
   - {doc_path}/Quick.md — шпаргалка (образец: Doc/Modules/signal_generators/Quick.md)

   Quick.md ОБЯЗАТЕЛЬНО начинается с раздела "## Концепция — зачем и что это такое":
   - Зачем нужен модуль (1-2 предложения, без формул и кода)
   - Для каждого класса/режима: "Что делает" + "Когда брать" — простыми словами
   - Аналогии, связи с другими модулями, ограничения
   - Цель: через полмесяца открыл файл → сразу вспомнил суть, не читая код

   Full.md содержит:
   - Математикой алгоритма (LaTeX)
   - Pipeline ASCII + mermaid диаграммой
   - C4 диаграммами (C1-C4)
   - C++ и Python примерами вызова
   - Таблицей тестов (C++ + Python) с развёрнутым описанием каждого теста:
     * Почему выбраны именно эти входные данные (синусоид, константа, линейный сигнал и т.д.)
     * Какое математически ожидаемое поведение (аналитический результат или CPU-эталон)
     * Какой конкретный баг/проблему ловит этот тест
     * Почему выбран именно такой порог (1e-3, 1e-4 и т.д.)
   - Файловым деревом модуля
   - Важными нюансами

⚠️ Правила работы с ссылками:
- ЗАПРЕЩЕНО вставлять ссылки на MemoryBank/tasks/* — эти файлы временные и будут удалены
- Всю полезную информацию из MemoryBank (планы, архитектурные решения, нюансы) — копировать ТЕКСТОМ прямо в Full.md
- В разделе «Ссылки» допустимы только: внутренние из Doc/* и Doc_Addition/*  файлы, внешние стабильные URL (GitHub, docs) 

После создания Full.md и Quick.md — **обязательно** создай или обнови `{doc_path}/API.md`:
- Используй команду `/api-doc $ARGUMENTS` как руководство
- API.md содержит: все публичные классы/методы с сигнатурами, цепочки вызовов, примеры C++ и Python
- Добавь ссылку на API.md в раздел "См. также" в Full.md и Quick.md

После этого обнови MASTER_INDEX.md если нужно.
