Добавь смысловые комментарии в файлы: $ARGUMENTS

Если аргумент не указан — спроси у пользователя.

## Что значит "смысловые комментарии"

НЕ это (бесполезно — очевидно из кода):
```cpp
int n;  // переменная n
void Process();  // метод Process
i++;    // инкремент i
```

А ЭТО — три вопроса: **ЧТО** (если неочевидно) + **ЗАЧЕМ** + **ПОЧЕМУ именно так**:
```cpp
// Размер FFT всегда степень двойки — hipFFT требует этого для оптимального плана.
// Округляем вверх через nextPow2(), чтобы не терять данные при усечении.
int fft_size;

// ВАЖНО: caller owns! Вызвать hipFree(result.data) после использования.
// Мы не освобождаем внутри — не знаем lifetime со стороны вызывающего кода.
OutputData Process(InputData input);
```

## Алгоритм

### Шаг 1: Определи что комментировать по аргументу

| Аргумент | Папка | Файлы |
|----------|-------|-------|
| `python` | `python/` | все `py_*.hpp` + `gpu_worklib_bindings.cpp` |
| `Python_test` | `Python_test/` | все `test_*.py` рекурсивно |
| `Python_test/filters` (с подпапкой) | `Python_test/filters/` | `test_*.py` в этой папке |
| `filters` (название модуля) | `modules/filters/` | `include/*.hpp`, `src/*.cpp`, `kernels/*.cl/.hip` |
| полный путь к файлу | файл напрямую | только этот файл |

Если аргумент не подходит ни под одно правило — спроси пользователя.

### Шаг 2: Прочитай контекст модуля

- Для C++ модуля: `Doc/Modules/$ARGUMENTS/Full.md` и `Quick.md`
- Для `python/`: все `Doc/Modules/*/Quick.md` (биндинги охватывают все модули)
- Для `Python_test/X/`: `Doc/Modules/X/Quick.md`
- Цель: понять роль в пайплайне ЦОС, ловушки, нюансы

### Шаг 3: Для КАЖДОГО файла — сначала READ и АНАЛИЗ, потом правка

**Обязательно прочитай файл полностью перед тем как что-то писать.**

После чтения составь мысленный список:
- Что уже прокомментировано (хорошо или поверхностно)?
- Что вообще не прокомментировано, но должно быть?
- Есть ли устаревшие/неверные комментарии?
- Какой стиль комментариев уже используется в файле? (продолжай в том же стиле)

Только после этого анализа — редактируй.

**Почему это важно:** файл может уже содержать часть комментариев (добавленных ранее агентом или вручную). Без чтения — риск дублирования или противоречия.

## Правила комментирования

### `.cpp` файлы реализации — совмещённый стиль: смысл + Doxygen

Эталон: `DrvGPU/backends/opencl/command_queue_pool.cpp`

⚠️ **ОБЯЗАТЕЛЬНЫЕ шаги для каждого `.cpp` файла:**

**ШАГ A — Section dividers** — если их нет, добавь перед каждой группой методов:
```cpp
// ════════════════════════════════════════════════════════════════════════════
// Инициализация
// ════════════════════════════════════════════════════════════════════════════
```

**ШАГ B — Doxygen-блок ПЕРЕД КАЖДЫМ методом** — это обязательно, не пропускать!
Структура: сначала смысловое объяснение (ЗАЧЕМ + ПОЧЕМУ), потом Doxygen теги:
```cpp
/**
 * @brief Краткое описание — ЧТО делает (одна строка)
 *
 * ЗАЧЕМ нужен + ПОЧЕМУ именно так:
 * - Ownership буферов: caller owns / внутри освобождается
 * - Порядок вызовов если важен
 * - Почему именно такой алгоритм (не альтернативный)
 *
 * Процесс (для сложных методов — нумеруй шаги):
 * 1. Что происходит первым
 * 2. Что происходит вторым
 *
 * @param param_name Что это, допустимые значения
 * @return Что возвращает
 * @note Важные ограничения, edge cases
 * @code
 * // Пример использования если неочевидно
 * @endcode
 */
void Method(Param p) {
```

Для простых методов (1-2 строки тела) минимальный вариант:
```cpp
/**
 * @brief ЧТО делает
 * ЗАЧЕМ / откуда берёт данные / что гарантирует
 * @return Что возвращает
 */
size_t GetCount() const {
```

**ШАГ C — Inline-комментарии внутри тела функции** — только нетривиальные места:
```cpp
void Cleanup() {
    // ROCm освобождается ПЕРВЫМ — намеренный порядок!
    // ZeroCopyBridge импортирует cl_mem в HIP: сначала разрушить OpenCL →
    // HIP получит dangling pointer → use-after-free в драйвере.
    rocm_->Cleanup();

    // clReleaseCommandQueue уменьшает счётчик ссылок; при 0 — очередь удаляется
    clReleaseCommandQueue(queue);
}
```

### `.hpp` заголовки — смысловой `//` без Doxygen тегов

Эталон: `DrvGPU/backends/opencl/command_queue_pool.hpp`

```cpp
// [Одна строка: ЧТО это и ЗАЧЕМ в пайплайне ЦОС]
// [ПОЧЕМУ именно так: ограничения, backend, связь с другими модулями]
class MyClass {
public:
    // ЗАЧЕМ метод нужен + что важно знать вызывающему:
    // - ownership буферов (caller owns / внутри освобождается)
    // - порядок вызовов (если важен)
    // - почему именно такой интерфейс (не альтернативный)
    ReturnType Method(Param p);

    int field_;  ///< ЗАЧЕМ это поле, почему такой тип/значение по умолчанию
```

### Python `.hpp` биндинги — смысловой `//` без Doxygen тегов

```cpp
// ЗАЧЕМ + ловушки для вызывающего (единицы, порядок, ownership):
// delay_us — МИКРОСЕКУНДЫ, не секунды! Число элементов = число антенн.
// При t < tau_i → сигнал антенны i = 0 (сигнал ещё не пришёл).
void set_delays(const std::vector<float>& delay_us);

GPUContext& ctx_;  ///< Ссылка — не владеет, контекст должен жить дольше объекта
```

### GPU буферы

```cpp
// ЗАЧЕМ буфер, размерность [A × B × C], тип элемента.
// Lifetime: кто создаёт, кто освобождает.
void* d_buffer_;  // [K × N], float2 — спектры K циклических сдвигов
```

### Kernel параметры и магические числа

```cpp
// ПОЧЕМУ именно 256 threads/block: оптимум для warp=64 на RDNA4
// При N < 256 — idle threads, при > 256 — register spill
constexpr int kBlockSize = 256;
```

## Что НЕ комментировать

- Очевидные вещи: `i++; // инкремент i`
- Уже хорошо объяснённые — не дублируй
- Временные отладочные строки — их надо удалить, не комментировать

## Что трогать, что нет

✅ Добавлять section dividers `════` в `.cpp` если их нет
✅ В `.cpp`: совмещённый блок — смысловой текст + Doxygen теги (@param, @return, @note, @code)
✅ В `.hpp` и Python биндингах: лаконичные `//` комментарии (без Doxygen тегов)
✅ Добавлять inline-комментарии внутри функций (нетривиальные блоки)
✅ Дополнять существующие комментарии если они поверхностные
✅ Исправлять устаревшие комментарии если не соответствуют коду
❌ НЕ менять сам код — только комментарии
❌ НЕ комментировать каждую строку — только там где неочевидно ЗАЧЕМ
